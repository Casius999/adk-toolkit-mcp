# 09 — A2A & MCP interop (the `a2a` and `mcp_bridge` domains)

Two interop surfaces: **A2A** (Agent-to-Agent — expose your agent as a service, or consume a remote
one) and the **MCP bridge** (expose ADK tools as MCP tool schemas). Maps to `a2a_*` and `mcp_bridge_*`.

## A2A: expose vs consume

```
Which direction?
├── EXPOSE my agent so others can call it over A2A
│     → a2a_expose      (writes a2a_app.py: a2a_app = to_a2a(root_agent, port=PORT))
│       and a2a_agent_card (build/inspect the AgentCard the served app advertises)
└── CONSUME a remote A2A agent inside my app
      → a2a_consume     (adds a RemoteA2aAgent proxy; compose it as a sub_agent)
```

**The `a2a` extra (`a2a-sdk`) is required to RUN any A2A surface** (each module imports `a2a.*`).
`a2a_consume` is codegen-only (no extra to ADD the proxy; the extra is needed to RUN `agent.py`).

## `a2a_consume` — proxy a remote agent
```
a2a_consume(path, app_name, name, agent_card_url)
```
Adds a `remote_a2a` agent (a `RemoteA2aAgent` proxy) and regenerates `agent.py`. `name` is the
generated variable; `agent_card_url` is the remote service's AgentCard URL (or a local JSON path). The
proxy has no children but can be composed as a `sub_agent` of another agent (`agents_compose`).

> **⚠️ Import path.** `RemoteA2aAgent` is **NOT** in `google.adk.agents` in 2.1.0. The generated code
> emits `from google.adk.agents.remote_a2a_agent import RemoteA2aAgent` (the only working import).
> `agent_card` is the 2nd positional arg.

## `a2a_expose` — serve your agent over A2A
```
a2a_expose(path, app_name, port=8001, execute=False)
```
Writes `<path>/<app_name>/a2a_app.py` (`a2a_app = to_a2a(root_agent, port=PORT)` + `from agent import
root_agent`). Then:
- **`execute=False`** (default, no extra) → returns `{file, serve_command, agent_card_path, executed:
  False}`. Serve it yourself: **`uvicorn a2a_app:a2a_app --host localhost --port <port>`** (run from the
  app dir so the sibling `agent` import resolves).
- **`execute=True`** (needs `a2a` extra) → starts a managed `uvicorn a2a_app:a2a_app` process via the
  same registry `dev_web` uses; returns `{key, pid, url, agent_card_url, executed: True}`. Drive it with
  `dev_status`/`dev_logs`/`dev_stop`.

Key A2A facts:
- `to_a2a(agent, *, host="localhost", port=8000, protocol="http", agent_card=None)` returns a
  **Starlette** app (decorated `@a2a_experimental` → a `UserWarning`, not a deprecation).
- **Routes register lazily on startup** (Starlette lifespan), so the well-known card route only appears
  once the server is actually running.
- The well-known AgentCard path is **`/.well-known/agent-card.json`** (extended card at
  `/agent/authenticatedExtendedCard`).
- A `str` `agent_card=` is a **file path**, not a URL.

## `a2a_agent_card` — build/inspect the AgentCard
```
a2a_agent_card(path, app_name, port=8001)   # async; needs the a2a extra
```
Imports `root_agent`, builds the card via `AgentCardBuilder(agent=…, rpc_url="http://localhost:<port>/")`
(async `build()`), and returns `card.model_dump(exclude_none=True)` (name/description/url/skills/
capabilities/…). `port` only composes the `rpc_url` written into the card (no server is started). Without
the `a2a` extra → actionable `err` (no crash).

## MCP bridge — expose ADK tools AS MCP tools (`mcp_bridge_*`)

This is the **reverse** of `tools_add_mcp_toolset` (which brings another MCP server's tools INTO ADK).
The bridge converts ADK tools to **MCP tool schemas** (`mcp.types.Tool`). **`mcp` is core** → no extra,
fully usable.

| Tool | Key args | Notes |
|---|---|---|
| `mcp_bridge_convert_builtin` | `kind` | Convert a **core builtin** by name (`google_search`, `url_context`, `load_memory`, `preload_memory`, `load_artifacts`, `get_user_choice`, `exit_loop`, `transfer_to_agent`, `enterprise_web_search`, `google_maps_grounding`) → `{name, description, inputSchema}`. **`vertex_ai_search` is rejected** (needs `data_store_id`) — attach it to an agent and use `expose_adk_tools` instead. |
| `mcp_bridge_expose_adk_tools` | `path, app_name, agent_name` | Import the project's `root_agent`, find `agent_name` (via `find_agent`, works for the root too), normalize its tools (`await canonical_tools()` wraps bare functions to `FunctionTool`), and convert each → `{count, tools: [{name, description, inputSchema}, ...]}`. An agent with no tools → empty list (not an error). Workflow agents (Sequential/Parallel/Loop) lack `canonical_tools` → guarded `err`. |

A builtin with no params (e.g. `google_search`) yields an **empty** `inputSchema {}`; a function tool
yields a real JSON-Schema (`{properties, required, type, title}`).

## When to use which interop

- Another team has an agent you want to call → `a2a_consume`.
- You want your agent callable by others (or by `adk web`'s A2A) → `a2a_expose` (+ `a2a_agent_card` to
  inspect what you advertise).
- You want to surface your ADK agent's tools to an MCP client → `mcp_bridge_expose_adk_tools`.
- You want another MCP server's tools available to your agent → `tools_add_mcp_toolset` (see `02-tools.md`).
